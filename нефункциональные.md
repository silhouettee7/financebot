
# Нефункциональные требования для распределенной системы «Финансовый бот»

## 1. Архитектурный контекст 
- **API** – публичный вход, роутинг запросов, пришедших от тг бота, HTTP.
- **App** – сценарии расходов/групп/копилки, HTTP к API.
- **БД (мастер)** – запись, **БД (чтение)** – аналитика.
- **Kafka** – асинхронные задачи (LLM, Excel, уведомления банка).
- **BankService** – OAuth2, синхронизация с банком (HTTP).
- **LLM‑сервис** – чтение из Kafka, запись в Redis, ответ через очередь.
- **Excel‑сервис** – чтение из Kafka, генерация отчётов, сохранение в Minio, ответ через очередь.
- **Minio** – объектное хранилище отчётов.
- **Redis** - key-value хранилище для ответов llm, кешированных запросов и диалогов

## 2. Общие

### 2.1 Доступность (Availability)
| Сервис | Целевой уровень | Примечание |
|--------|----------------|-------------|
| API | **99.9 %** | Критический путь |
| App | **99.5 %** | Основные бизнес-сценарии |
| BankService (синхронизация) | **99.0 %** | Влияет на актуальность данных |
| LLM / Excel (асинхронные) | **95.0 %** | Допустима задержка, долгая асинхронная операция |

- **Graceful degradation**: при отказе LLM/Excel/BankService – сообщения идут в кафку, откуда впоследствии обработются при поднятии сервисов, но расходы принимаются. Обратная ситуация в случае падения всех App, LLM/Excel/BankService работают, но ручное внесение трат, копилка и тд не работают. API должен работать в идеале всегда без перебоев, поэтому максимально безопасный в плане выполнения операций (только роутит и читате с редиса и минио)
- **Отсутствие SPOF главных компонент**: минимум 2 реплики API, 3 брокера Kafka, репликация БД.

### 2.2 Производительность и масштабируемость
| Показатель | Цель (SLO) | Допустимый максимум |
|------------|------------|----------------------|
| **RPS общий (API)** | 450 RPS | 1000 RPS (с rate limiting) |
| **RPS App** | 200 RPS | 500 RPS |
| **RPS LLM запросов** | 60 RPS | 700 RPS (через очередь) |
| **RPS Excel** | 50 RPS | 700 RPS |
| **RPS BankService** | 140 RPS | 700 RPS |

**Масштабирование:**
- API, App – горизонтальное.
- LLM/Excel/BankService consumer – увеличение партиций Kafka + реплик consumer.
- БД – master для записи, replica(s), аналитическая для отчётов/LLM/Excel.

### 2.3 Latency для пользователя (весь путь) – целевые перцентили

| Операция | P50 | P95 | P99 | Примечание |
|----------|-----|-----|-----|-------------|
| **Запись расхода** (синхронно) | < 50 мс | < 200 мс | < 500 мс | Критично для UX |
| **Настройка групп / копилки и тп** | < 50 мс | < 150 мс | < 300 мс | Легкие сценарии |
| **LLM вопрос** | < 6 с | < 11 с | < 25 с | очередь + обработку + кеш|
| **Генерация Excel‑отчёта** | < 3 с | < 7 с | < 11 с | Minio запись + генерация  + очередь|
| **Синхронизация с банком**  | < 1 с | < 2 с | < 5 с | OAuth |
| **Синхронизация трат банком (уведомление) - ручной вызов** | < 7 с | < 12 с | < 26 с | очередь + запрос в банк + БД |

### 2.4 Согласованность (Consistency)
- **Сильная согласованность** – запись расхода (пользователь сразу видит свой расход).
- **Eventual consistency** – отчёты и графики (допустимо).
траты с банка синхронятся автоматически или по запросу пользователя, однако уведомление о том, что траты есть и их нужно рапределить приходит по факту их записи в БД, так что согласованность соблюдается

### 2.5 Безопасность (Security)
- **OAuth2** для синхронизации с банком.
- **Шифрование токенов банка**.
- **Vault или аналог для хранения API ключа для банка и других секретов**.

## 3. Надёжность и устойчивость к отказам

### 3.1 Kafka и очереди
| Механизм | Требование |
|----------|-------------|
| **DLQ (Dead Letter Queue)** | Для каждого consumer топика (`llm-dlq`, `excel-dlq`, `bank-dlq`). При неудачных попытках – в DLQ + лог |
| **Outbox pattern** | запись в Postgres outbox + фоновая пересылка в случае BankService |

**Поведение при падении Kafka:**
- API возвращает - сервис недоступен.
- Сообщение сохраняется в **outbox‑таблице** (БД на запись) - в случае BankService.
- Воркер пытается переслать в Kafka.


### 3.2 Отказоустойчивость сервисов
- **Retry + Exponential backoff** например для BankService.
- **Health checks** (liveness/readiness) для всех сервисов в оркестраторе.
- **Резервное копирование БД** каждые 4 часа

| `POST /sync` | 1 запрос/10 сек | `429` |


## 6. Выбор протоколов: HTTP и gRPC

| Взаимодействие | Протокол | Обоснование |
|----------------|----------|--------------|
| API ↔ App | HTTP| Простота, REST, не требуется стриминг, начальный RPS не большой|
| API ↔ BankService | HTTP | OAuth2 редиректы удобнее через HTTP. |
| Внутренние (Kafka, Minio) | родной протокол | Стандарт. |
| *Перспектива* (при росте) | gRPC для API↔App | Когда RPS > 5000 или строгие контракты станут критичны. |
